# mHost 功能分解与阶段规划大纲

创建日期：2026-06-18

本文基于项目 `readme.md` 中对 mHost 的定位，并参考 SwitchHosts、Gas Mask、HostsMan、Hosty 等 hosts 管理工具的公开资料，整理 mHost 的功能边界、模块拆分和阶段规划。本文是后续 PRD、技术方案和任务拆分的上游大纲，不直接替代详细需求文档。

## 1. 产品定位

mHost 应定位为一个轻量、快速、低打扰的跨平台 hosts 与域名解析管理工具，核心价值不是“更好看的 hosts 编辑器”，而是降低环境切换、规则管理和广告屏蔽的操作成本。

从竞品看，传统 hosts 管理工具普遍围绕“多份 hosts 文件管理、一键启用、远程规则更新、备份恢复、错误修复、系统托盘切换”展开。SwitchHosts 明确支持 system/local/remote/group/folder 等 hosts 条目管理、主窗口或系统托盘快速切换、语法高亮、查找替换、远程 hosts 手动/定时/启动刷新、导入导出、回收站、写入模式、代理、启动项、应用后命令和本地 HTTP API 等能力。<sup><a href="#cite-1">[1]</a></sup> Gas Mask 则以 macOS 菜单栏和编辑器为中心，支持 Local、Remote、Combined 三类 hosts 文件，并通过激活文件更新 `/etc/hosts`。<sup><a href="#cite-2">[2]</a></sup> HostsMan 更偏 Windows 工具属性，强调 hosts 更新器、自定义来源、内置编辑器、错误修复、故障诊断、备份恢复、启停 hosts、压缩包支持、排除列表和 hostname resolver。<sup><a href="#cite-3">[3]</a></sup>

mHost 的差异点应围绕 README 已确定的方向展开：默认不依赖超级用户授权、跨平台体验一致、运行轻量、规则可理解、广告屏蔽与开发环境隔离。竞品多数仍以修改系统 hosts 文件为主，Gas Mask 首次启动会请求密码，因为它需要提升权限来修改 `/etc/hosts`。<sup><a href="#cite-2">[2]</a></sup> 这说明“免超级用户授权模式”可以成为 mHost 的产品和技术分界线，但也会带来本地 DNS、代理解析或系统网络配置兼容性问题，需要单独拆成阶段推进。

## 2. 竞品能力摘要

| 工具 | 平台与定位 | 关键能力 | 对 mHost 的启发 |
| --- | --- | --- | --- |
| SwitchHosts | 基于 Tauri 的 hosts 文件管理应用；公开 README 显示使用 Tauri、React、Jotai、Mantine 等技术 | 多类型 hosts 条目、快速切换、系统托盘、语法高亮、查找替换、远程更新、导入导出、回收站、偏好设置、本地 HTTP API | mHost 的基础能力不能只覆盖 Profile CRUD，还需要规则来源、快速入口、编辑效率、备份恢复和扩展接口 |
| Gas Mask | macOS hosts 文件管理器 | Local、Remote、Combined 文件；菜单栏快速切换；后台运行；激活后写入 `/etc/hosts` | “组合规则”值得吸收，适合把开发环境规则和广告屏蔽规则分层叠加 |
| HostsMan | Windows freeware hosts 管理工具 | 内置更新器、自定义来源、错误修复、故障诊断、备份管理、启停 hosts、排除列表、hostname resolver | Windows 用户更需要诊断、修复、备份、回滚和安全提示；广告屏蔽规则来源管理也应独立成模块 |
| Hosty | 公开资料有限，下载站描述其用于创建、管理、导入和导出 hosts 文件 | 创建、管理、导入、导出 | 只能作为基础能力佐证；不应基于它设计高级能力 |

## 3. 功能模块分解

### 3.1 Profile 管理

Profile 是 mHost 的核心对象，用来承载一组域名解析规则和使用场景。

应包含：

- Profile 创建、编辑、复制、删除。
- Profile 启用、暂停、切换。
- Profile 名称、说明、标签、分组、颜色或图标标识。
- Profile 只读状态，用于保护生产环境或外部订阅规则。
- Profile 排序、搜索、过滤。
- Profile 导入、导出。
- Profile 修改历史和恢复。

需要后续细化的问题：

- 是否允许多个 Profile 同时启用。
- 多个 Profile 同时启用时，冲突规则如何处理。
- 是否需要“默认 Profile”或“安全 Profile”。
- 是否需要 Profile 模板，例如 Development、Testing、Staging、Production、Ad Block。

### 3.2 Hosts 规则编辑

规则编辑能力决定开发者是否愿意长期使用 mHost。

应包含：

- 支持标准 hosts 语法解析。
- 支持 IP、域名、注释、空行。
- 支持启用或禁用单条规则。
- 支持批量启用、暂停、删除。
- 支持语法高亮。
- 支持查找和替换。
- 支持重复域名、无效 IP、非法域名、格式错误提示。
- 支持规则备注。
- 支持规则分组。
- 支持冲突检测。
- 支持复制为 hosts 文本。

需要后续细化的问题：

- 编辑器优先做“文本编辑器”还是“表格化规则编辑器”。
- 是否允许两种视图切换。
- 是否需要保留原始注释和格式。
- 是否支持 IPv6。

### 3.3 规则应用引擎

规则应用引擎负责把 Profile、广告规则、白名单、远程规则合并成最终生效结果。

应包含：

- 当前生效规则预览。
- Profile 规则合并。
- 广告屏蔽规则合并。
- 白名单覆盖。
- 冲突规则排序。
- 规则来源追踪。
- 应用前检查。
- 应用后状态确认。
- 应用失败回滚。
- DNS 缓存刷新提示或自动处理。

需要后续细化的问题：

- 规则优先级是否采用固定顺序：白名单 > 手动 Profile > 团队/远程 Profile > 广告屏蔽。
- 是否允许用户自定义优先级。
- 应用失败时是否自动恢复上一版。

### 3.4 免超级用户授权模式

这是 mHost 区别于传统 hosts 管理器的关键模块。竞品通常直接修改系统 hosts 文件；Gas Mask 也明确需要提升权限来修改 `/etc/hosts`。<sup><a href="#cite-2">[2]</a></sup> mHost 应把免授权模式作为默认目标，同时提供清楚的兼容性说明。

可拆为三种实现路径：

- 本地 DNS 模式：在本机启动轻量 DNS 服务，按 mHost 规则解析域名。
- 代理解析模式：通过本地代理处理浏览器或应用请求。
- 系统 hosts 写入模式：作为兼容 fallback，需要用户明确授权。

应包含：

- 当前工作模式展示。
- 工作模式切换。
- 本地 DNS 健康检查。
- 系统 DNS 配置检测。
- 端口占用检测。
- 权限请求原因说明。
- 失败时回退到安全状态。

需要后续细化的问题：

- MVP 是否必须实现免授权模式，还是先用系统 hosts 写入模式验证产品流。
- macOS 和 Windows 的本地 DNS 模式是否可以保持一致体验。
- 普通用户广告屏蔽场景是否默认走本地 DNS。

### 3.5 远程规则与订阅

SwitchHosts 和 Gas Mask 都支持远程 hosts；SwitchHosts 支持手动、定时或启动时刷新远程 hosts。<sup><a href="#cite-1">[1]</a></sup><sup><a href="#cite-2">[2]</a></sup> HostsMan 也提供内置 updater 和自定义本地或远程来源。<sup><a href="#cite-3">[3]</a></sup>

应包含：

- 添加远程 hosts URL。
- 远程规则手动刷新。
- 启动时刷新。
- 定时刷新。
- 刷新失败重试。
- 远程规则只读。
- 远程规则本地缓存。
- 远程规则版本记录。
- 远程规则来源可信提示。
- 远程规则导入到 Profile。

需要后续细化的问题：

- 是否内置常用广告屏蔽规则源。
- 远程规则是否允许用户编辑本地覆盖项。
- 远程规则失败时使用旧缓存还是禁用该来源。

### 3.6 广告屏蔽

README 已把广告屏蔽列为核心场景。该模块不应与开发环境 Profile 混在一起，而应作为可独立启停、可叠加的规则层。

应包含：

- 广告屏蔽总开关。
- 规则源管理。
- 手动添加屏蔽域名。
- 白名单。
- 规则更新。
- 命中来源展示。
- 屏蔽规则与开发规则冲突提示。
- 广告屏蔽 Profile 化。
- 屏蔽效果状态页。

需要后续细化的问题：

- 是否支持 EasyList 一类非 hosts 格式规则，还是只支持 hosts 格式。
- 是否需要“追踪屏蔽”“广告屏蔽”“恶意域名屏蔽”分类。
- 命中统计只有在本地 DNS 或代理模式下才可靠，系统 hosts 模式可能无法提供实时命中。

### 3.7 系统托盘与快速切换

SwitchHosts 和 Gas Mask 都强调主窗口之外的快速切换入口。<sup><a href="#cite-1">[1]</a></sup><sup><a href="#cite-2">[2]</a></sup> mHost 应把低打扰体验落实到托盘和菜单栏设计。

应包含：

- 托盘/菜单栏常驻入口。
- 当前启用 Profile 展示。
- 快速切换 Profile。
- 快速启停广告屏蔽。
- 打开主窗口。
- 刷新远程规则。
- 状态异常提示。
- 退出应用。

需要后续细化的问题：

- 是否允许完全不常驻后台。
- 常驻后台时的资源占用目标是多少。
- 托盘状态是否显示规则应用状态、DNS 状态或错误状态。

### 3.8 备份、恢复与回滚

HostsMan 把备份管理作为主要功能，SwitchHosts 也支持导入导出和回收站。<sup><a href="#cite-1">[1]</a></sup><sup><a href="#cite-3">[3]</a></sup> mHost 需要把“可恢复”作为安全感的一部分。

应包含：

- 手动备份。
- 自动备份。
- 修改前备份。
- 恢复到上一版。
- 导出全部配置。
- 导入配置。
- 删除后进入回收站。
- 清空回收站。
- 配置迁移。

需要后续细化的问题：

- 自动备份保留多少份。
- 是否备份远程规则缓存。
- 是否备份偏好设置和历史记录。

### 3.9 诊断与修复

HostsMan 提供错误修复、故障诊断、重复项处理和 hostname resolver。<sup><a href="#cite-3">[3]</a></sup> mHost 应将诊断作为减少用户困惑的关键能力。

应包含：

- hosts 语法检查。
- 重复规则检查。
- 冲突规则检查。
- 当前域名解析测试。
- hostname resolver。
- DNS 缓存状态提示。
- 本地 DNS 服务状态检查。
- 端口占用检查。
- 系统 hosts 文件权限检查。
- 常见问题修复建议。

需要后续细化的问题：

- 是否提供“一键修复”。
- 修复前是否必须创建备份。
- 诊断结果是否允许导出给技术支持。

### 3.10 偏好设置与扩展

SwitchHosts 的偏好设置覆盖写入模式、代理、更新检查、开机启动、应用后命令和本地 HTTP API。<sup><a href="#cite-1">[1]</a></sup> mHost 可以分阶段吸收其中适合自身定位的能力。

应包含：

- 开机启动。
- 主题和语言。
- 写入模式或解析模式设置。
- 代理设置。
- 远程规则刷新策略。
- 应用后命令。
- 本地 API。
- 数据存储位置。
- 日志级别。

需要后续细化的问题：

- 本地 API 是否放到后期。
- 是否支持 CLI。
- 是否支持团队共享配置。

## 4. 阶段规划

### 阶段 0：产品骨架与数据模型

目标是把 mHost 的核心概念稳定下来，避免后续功能不断返工。

建议范围：

- 明确 Profile、规则、规则源、广告屏蔽、白名单、应用状态的数据模型。
- 明确本地存储结构。
- 明确多 Profile 合并规则。
- 明确跨平台应用壳。
- 明确系统 hosts 写入模式与免授权模式的关系。
- 形成第一版信息架构。

阶段产出：

- 数据模型草案。
- 产品导航结构草案。
- Profile 与规则合并逻辑草案。
- 权限策略说明。

不建议纳入：

- 完整广告屏蔽。
- 团队同步。
- 本地 API。
- 复杂诊断。

### 阶段 1：MVP，可用的 Profile 切换

目标是让开发者能用 mHost 管理多环境 hosts 并完成可靠切换。

建议范围：

- Profile 创建、编辑、删除。
- 单 Profile 启用和切换。
- hosts 文本编辑。
- 基础语法检查。
- 当前生效状态展示。
- 应用前备份。
- 应用失败回滚。
- 导入和导出 Profile。
- macOS 和 Windows 的基础适配。

阶段产出：

- 可创建 Development、Testing、Staging、Production 等 Profile。
- 可一键切换当前 Profile。
- 可从已有 hosts 文本导入规则。
- 可恢复上一版配置。

关键验收方向：

- 切换后用户能判断当前启用了哪个 Profile。
- 规则格式错误不会直接写入生效配置。
- 应用失败不会破坏原系统 hosts。

### 阶段 2：编辑效率与安全感

目标是让 mHost 从“能用”变成“愿意长期使用”。

建议范围：

- 规则搜索。
- 查找替换。
- 语法高亮。
- 规则启停。
- 规则分组和标签。
- 冲突检测。
- 重复规则检测。
- 回收站。
- 备份管理。
- 修改历史。
- 系统托盘或菜单栏快速切换。

阶段产出：

- 复杂 hosts 文件可维护。
- 删除和覆盖操作可恢复。
- 常用 Profile 可从托盘快速切换。

关键验收方向：

- 大量规则下编辑不卡顿。
- 用户可以快速定位某个域名来自哪个 Profile。
- 误删、误改可以恢复。

### 阶段 3：远程规则与广告屏蔽

目标是支持广告屏蔽和远程规则订阅，同时不污染开发环境配置。

建议范围：

- 远程 hosts URL 添加。
- 手动刷新远程规则。
- 启动时刷新。
- 定时刷新。
- 远程规则缓存。
- 广告屏蔽总开关。
- 广告规则源管理。
- 白名单。
- 规则来源展示。
- 远程规则失败提示。

阶段产出：

- 用户可启用广告屏蔽规则。
- 用户可添加和刷新远程 hosts 订阅。
- 白名单可以覆盖广告屏蔽。
- 开发 Profile 与广告屏蔽规则分层管理。

关键验收方向：

- 广告屏蔽不会覆盖开发环境规则。
- 远程规则不可用时不会导致解析状态混乱。
- 用户能看懂某条规则来自手动配置、远程订阅还是广告屏蔽。

### 阶段 4：免超级用户授权模式

目标是兑现 README 中“优先采用不直接修改系统 Hosts 文件”的关键承诺。

建议范围：

- 本地 DNS 模式原型。
- 本地 DNS 健康检查。
- 系统 DNS 配置检测。
- 端口占用检测。
- 规则应用到本地 DNS。
- 模式切换：本地 DNS、代理解析、系统 hosts 写入。
- 权限请求最小化说明。
- 失败回退策略。

阶段产出：

- 用户可在不修改系统 hosts 文件的情况下使用核心解析能力。
- 需要系统权限的操作有明确原因和边界。
- 应用能检测当前解析是否按 mHost 规则工作。

关键验收方向：

- 无管理员权限时仍能完成主要域名解析场景。
- 本地 DNS 异常时能恢复系统默认网络状态。
- 用户能理解当前模式和影响范围。

### 阶段 5：诊断、扩展与高级能力

目标是提升复杂场景下的可靠性，并为团队、自动化和外部工具留下扩展空间。

建议范围：

- hostname resolver。
- 域名解析测试。
- DNS 缓存刷新。
- 诊断报告。
- 应用后命令。
- 本地 HTTP API。
- CLI。
- 团队配置共享。
- 配置备份同步。
- 规则源推荐。

阶段产出：

- 用户可以自查“为什么没生效”。
- 高级用户可以把 mHost 接入开发脚本。
- 团队可以共享标准环境配置。

关键验收方向：

- 诊断结果能定位到规则、Profile、DNS、权限或系统缓存层面的原因。
- API 和 CLI 不破坏低打扰定位，默认关闭或明确授权。
- 团队共享不会覆盖个人本地配置。

## 5. 建议优先级

| 优先级 | 功能方向 | 原因 |
| --- | --- | --- |
| P0 | Profile 管理、规则编辑、切换应用、基础备份、状态展示 | 这是 mHost 的最小可用闭环 |
| P0 | 规则解析、校验、冲突检测 | 直接影响安全性和可信度 |
| P1 | 托盘快速切换、搜索、语法高亮、导入导出、回收站 | 提升日常使用效率 |
| P1 | 远程规则、广告屏蔽、白名单 | 对应 README 中的广告屏蔽场景 |
| P1 | 免超级用户授权模式原型 | 对应 mHost 的核心差异点，但技术风险较高 |
| P2 | 诊断、resolver、应用后命令、本地 API、CLI | 面向高级用户和复杂场景 |
| P2 | 团队共享、配置同步 | 有价值，但不应影响单机工具的轻量定位 |

## 6. 后续需要澄清

1. MVP 是否必须坚持“免超级用户授权”，还是可以先用系统 hosts 写入模式验证核心体验。
2. mHost 第一阶段是否同时支持 macOS 和 Windows，还是先选择一个主平台。
3. 编辑器形态优先文本编辑、表格编辑，还是双视图。
